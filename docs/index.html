about

Okay... but why??

aspLite is a lightweight framework for ASP/VBScript developers. It can help to develop better ASP/VBScript applications. aspLite does not reinvent, replace or rewrite ASP components. It only tries to come up with a new way of using some of them. ASP/VBScript (also known as "classic ASP") is available for 20 years now and I believe it still makes sense to share this approach. After all, classic ASP has proven to be reliable, fast and secure. And no, classic ASP is not dead. That's fake news, instantiated by internet trolls.

But that's not all. Today's web applications are using JavaScript frameworks (Angular, Vue, React, jQuery, etc). Only data is transmitted back and forth to the server. That's why a server-side technology better stays small and simple. Hence the name "aspLite".

Some examples perhaps. ASP developers never had a native way to create zip files on the server, unless they used 3rd party DLLs (or buggy shell-scripts). That was not very convenient. The aspLite demo site ships with a copy of JSZip where a zip file is created in the browser (with files residing on the server). Same story for PDF files: jsPDF. The vast majority of these JavaScript plugins are free to use, backed by 100's of contributors and they work in all (current) browsers, on all devices. How amazing is that?! Ever since the release of HTML5 and JavaScript5 (somewhere between 2009 and 2012), browser-sided JS/CSS/HTML frameworks have become very popular. Much more popular than their server-sided predecessors (ASP(.NET), PHP, etc). The most starred repositories on GitHub are all about (learning) JavaScript.

The aspLite demo puts the following front-end HTML/CSS/JS frameworks/plugins at work: Bootstrap, jQuery, JSZip, CkEditor, CodeMirror, jsPDF and DataTables. These are just a few of the many that could be used in web applications nowadays.

developer guidelines


Installation

Install as a website or virtual directory in IIS. Make sure IUSR has full permissions on the entire folder and subfolders. aspLite does not require any 3rd party dll (COM) to be installed.
The actual framework is located in /asplite/asplite.asp. The path /asplite can be changed to anything else in /asplite/config.asp. If you prefer to use aspLite in a subfolder /mypath/asplite, make sure to update config.asp correspondingly. To run the demo site that ships with the framework, you need to enable 32bits applications for your website's application pool, as it includes an Access database sample.

aspLite can also be used in any IIS Express edition.

Less is more

aspLite consists out of 2 asp files only:

/asplite/asplite.asp
/asplite/config.asp

The only thing this framework really does, is wrap useful ASP components into one reusable VBScript class:

* split html/css/javascript and asp
* conditionally load asp code (rather than include them 'all')
* .net-alike codebehind approach
* charset utf8-proof
* file streaming (text, code + binary)
* ServerXMLHTTP & DOMDocument
* basic AJAX support
* application caching 
* catch 404 requests
* some generic ASP/VBScript functions
* write your own plugins

Get started

You may want to check demo.asp. It helps to understand how aspLite works.

To start using aspLite, include it at the very top of your asp-page:
<!-- #include file="asplite/asplite.asp"-->

From then on, you can use an object of the aspLite class: aspL (it's already created). The word "aspL" should therefore be considered a reserved word. Do not re-declare, re-set or even destroy it. 

Stick to 1 asp file for your project (Single-Page Applications)

There are some very good reasons to stick to 1 asp page only for your application, even if your application has 20 screens with various different menus, forms, buttons or links each. Your best option is to use "default.asp" in the root of your application. "Default.asp" is always included in the list of "default documents" in IIS (Express). Another advantage of Single Page Applications, is that HTML/CSS/Scripts are only loaded once throughout the lifespan of an application. Only data is transmitted back and forth. On top of that, using 1 asp page only, keeps server memory usage very low, specially when combined with conditionally loading asp scripts.

Catch 404 requests

There is another reason to stick to 1 asp page for your app: configure that page to execute 404 request. aspLite ships with a function (aspLite.pathinfo) that retrieves any custom url (pointing to non-existing items). Make sure you add this to web.config (in the root of your site):
(thanks to Sergio Conselvan for providing a copy that works for both IIS & IIS Express)

<?xml version="1.0"?>
    <configuration>
        <system.webServer>
            <httpErrors errorMode="Custom">
                    <remove statusCode="404" subStatusCode="-1" />
                    <error statusCode="404" path="/default.asp" responseMode="ExecuteURL" />
            </httpErrors>
        </system.webServer>
    </configuration>

(change /default.asp to the page of your choice - eventually add the virtual directory, eg: /mydir/default.asp)

The advantage of using this technique, is that your urls would look like:
www.site.com/folder (rather than www.site.com/default.asp?folder=foldername)
www.site.com/folder/folder
www.site.com/folder/folder /folder
Not only this looks better and it's easier to type, you can also retrieve a lot of information from these 404 requests. This is a very powerful technique, available to classic ASP developers for years already (as from Windows 2000). If you use this technique consistently, nobody will ever be able to tell which technology your site uses (php? asp? .net?). This technique is not harming SEO, as status 200 is returned to the browser. It's even improving SEO a lot, as userfriendly (and meaningful) urls are used.

Resx files?

As an ASP developer, you will quickly notice that the .resx extension is used to include ASP scripts. Why? For security reasons. Resx files are never served nor executed by IIS. Therefore, it's impossible to inject or open resx files in the browser directly. You *can* however also use the regular .asp extensions for your handler-files. Make sure that visitors cannot inject (querystring, form submit) or load these pages in the browser directly.

IDE

Classic ASP developers have been lacking a dedicated IDE (Integrated Development Environment) for some time now, unless they don't mind to wait 30 seconds for some bloated .NET monster raises from the dead (and eats all your ram and kills your cpu). I therefore prefer to use Notepad++. It's free, lightning fast, reliable and it's very easy to enable ASP syntax highlighting for .resx files. Open Settings, Style configurator, select "asp", add "resx" to the "user ext". You have to reopen the resx files for this to take effect. Notepad++ also comes with some basic code-completion functions. Others may want to use Visual Studio Code instead.

JScript?

In the early days, classic ASP was also developed in JScript, the Microsoft variant of JavaScript back then. JScript never reached the level of popularity VBScript did when it came to classic ASP. aspLite is 100% about VBScript and as such not useful for JScript developers. But demo.asp includes a little JScript sample, via server.execute(). Not sure anyone would ever want to use it this way.

plugins

aspLite will always have a very small footprint on your application and on the server. You can however develop your own plugins for aspLite. How?

The following plugins are included already - but not necessary to use the framework i.e. they can be ignored. When no plugins are used, they will not affect security or performance of your application in any way. Demo.asp uses all plugins available:

* connector for Access databases (OLEDB)
* mailer for CDO.Message 
* jpg resize/recolor/crop (.net required) 
* text/number randomizing 
* rss feed reader
* aspJSON by rcdmk
* file uploader (based on freeaspupload.net)
* Secure Hash Algorithm (SHA256)
* MD5 (Message Digest Algorithm 5) 

How to add your own?

Let's inspect one of the example plugins: asplite/plugins/helloworld/helloworld.resx

<%
class cls_asplite_helloworld

    public function hw
        hw="Hello world!"
    end function

end class
%>

Some remarks:

* You have to use the resx-extension for your main plugin-file, for security reasons (see above).

* The entire codeblock sits beteween <% and %> code render blocks. This is not required, but it can help to trigger syntax highlighting, like when working in Notepad++. Be aware that at runtime, aspLite removes code render blocks - see asplite.exec(path)

* The entire content of this resx file will be considered ASP code. Do not mix it with html, text, js or css code. It would raise an error.

* Even though I always use Option Explicit (forces to declare variables only once), it seems to have no effect on code that is executed through VBScript's executeGlobal. I have no idea why, but it's good to know I guess.

* As aspLite forces to use the resx-extension, your plugin cannot include other ASP files like you may be used to: 

<!-- #include file="mypluginInc.asp"--> would raise an error, even if that mypluginInc.asp exists.

You would need to use the aspLite conditional include instead:

aspL.exec("asplite/plugins/myplugin/mypluginInc.resx")

where mypluginInc.resx resides in your "myplugin" directory.

* Plugin-folders can include other subfolders and files, like html, js, css, images, etc. There are no limitations whatsoever.

These are the steps to create a plugin for aspLite:

1

Locate

Plugins are stored in /asplite/plugins

2

Add your plugin-folder

Add a seperate folder for your plugin named myplugin: eg: /asplite/plugins/myplugin/

3

Create plugin-resx

Add a resx file with the same name: /asplite/plugins/myplugin/myplugin.resx. 

4

Create class

In myplugin.resx, create a class with this same name as follows:

class cls_asplite_myplugin
...
end class

5

Usage

To create an instance of your plugin somewhere in your application, use:

dim myObj
set myObj=aspL.plugin("myplugin")
myObj.XX()

You could also just:
aspL.plugin("myplugin").XX()

(where XX() is a public sub/function of your myplugin-class)
